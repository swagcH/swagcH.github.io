1.什么是JUC

JUC（Java Util Concurrency）是Java工具类库中的一部分，提供了在多线程环境下编写并发程序所需的工具和框架。JUC包含了一些并发编程的核心概念和工具，使得开发人员能够更轻松地编写高效、线程安全的并发程序。

Synchronized 和 Lock 区别

  1. synchronized关键字和Lock接口是Java中用于实现线程同步的两种不同方式，它们之间有以下几点区别：

    1. 使用方式：

      • synchronized是Java语言提供的关键字，可以直接用于方法或代码块中，不需要显式地创建锁对象。
      • Lock是一个接口，需要显式地创建锁对象，通常使用ReentrantLock类来实现。

    2. 灵活性：

      • synchronized是一种内置的锁机制，比较简单，但缺乏一些高级功能。
      • Lock接口提供了更多的功能，如可中断锁、超时获取锁、尝试非阻塞获取锁等。

    3. 等待锁的方式：

      • synchronized在获取不到锁时，线程会进入阻塞状态，直到获取到锁为止。
      • Lock在获取不到锁时，可以选择等待、放弃等待或者做其他处理。

    4. 释放锁的方式：

      • synchronized在方法执行完毕或代码块结束时会自动释放锁。
      • Lock需要手动调用unlock()方法来释放锁。

    5. 性能：

      • synchronized在JVM层面上对锁进行优化，因此在某些情况下性能可能比Lock更好。
      • Lock提供了更多的灵活性和控制，但由于是通过Java代码实现的，性能上可能会稍差。

    总的来说，synchronized适用于简单的线程同步需求，而Lock适用于对锁需要更精细控制和灵活性的场景。在选择使用时，需要根据具体情况来决定哪种方式更合适。

虚假唤醒问题

虚假唤醒（Spurious Wakeup）是指在多线程编程中，一个线程在没有被通知、没有收到信号的情况下，从等待状态（通常是wait()函数）中被唤醒，但此时并没有符合其等待条件的事件发生。这种情况可能会在某些多线程环境中发生，尽管并没有其他线程调用notify()或notifyAll()来通知等待线程，等待线程仍然被唤醒。

Cow,读写分离

Copy-on-write（写时复制）是一种用于处理共享数据的技术，它可以解决写入时覆盖的问题。当多个进程或线程要访问同一个资源时，如果每个进程都复制一份资源的副本，会占用大量内存。相反，可以让所有进程共享同一个资源的副本，但只有在有进程要修改资源时才会复制资源，这就是写时复制的概念。

具体来说，当一个进程要修改资源时，系统会首先检查资源的引用计数。如果这个资源只被一个进程引用，那么该进程可以直接修改资源而无需复制。但如果资源被多个进程引用，系统就会将资源复制一份，然后让要修改资源的进程修改新复制的资源，而其他进程继续共享原始资源。这样一来，每个进程都可以独立地修改自己的资源副本，而不会影响其他进程。

通过写时复制，可以在多个进程之间共享资源的同时，避免了写入时的覆盖问题，提高了系统的性能和效率。

原理：

ConcurrentHashMap 是 Java 中线程安全的哈希表实现，它采用了一种不同于传统 HashMap 的机制来实现并发访问。主要原理如下：

  1. 分段锁（Segment）：ConcurrentHashMap 内部维护了一个 Segment 数组，每个 Segment 就是一个独立的哈希表，类似于将一个大的哈希表分成多个小的哈希表。每个 Segment 都相当于一个小的 HashMap，它们之间相互独立，可以独立进行并发操作。

  2. Hash 表的扩容：与 HashMap 类似，ConcurrentHashMap 也会在需要时对哈希表进行扩容。但不同的是，它只会对某个 Segment 进行扩容，而不是对整个哈希表进行扩容。这样可以减少扩容时的锁竞争，提高并发性能。

  3. put 操作的实现：当执行 put 操作时，ConcurrentHashMap 首先根据 key 的哈希值确定要放入哪个 Segment 中，然后在该 Segment 上加锁，进行插入操作。这样可以保证在放入元素时，其他线程可以继续访问其他 Segment 中的元素，不会受到阻塞。

  4. get 操作的实现：与 put 操作类似，get 操作也会根据 key 的哈希值确定要查找哪个 Segment，然后在该 Segment 上加锁，进行查找操作。这样可以保证在查找元素时，其他线程可以继续访问其他 Segment 中的元素，不会受到阻塞。

  5. 性能优化：ConcurrentHashMap 在实现上进行了多项性能优化，如使用了 volatile 修饰的数组元素来保证可见性，使用了 CAS（Compare and Swap）操作来保证原子性，以及使用了分段锁来减少锁竞争等。

四种拒绝策略

在 Java 中，线程池中的任务队列满了，无法继续接受新任务时，线程池会根据配置的拒绝策略来处理这种情况。常见的四种拒绝策略如下：

  1. AbortPolicy（默认策略）：直接抛出 RejectedExecutionException 异常，表示拒绝执行新任务。

  2. CallerRunsPolicy：将任务交给调用线程来执行，如果线程池已满，新任务将在调用线程中执行。这样可以降低新任务的提交速度，避免任务堆积，但可能会影响调用线程的性能。

  3. DiscardPolicy：直接丢弃无法处理的任务，不做任何处理。这种策略可能会导致任务丢失，慎用。

  4. DiscardOldestPolicy：丢弃最早加入任务队列的任务，然后尝试再次提交新任务。这种策略可以保证新任务能够得到处理，但会丢失一部分任务。

如何去设置线程池的大小

判断当前系统属于IO密集型还是CPU密集型

  1. CPU 密集型任务：对于 CPU 密集型任务，线程池的大小应该设置为接近或等于处理器核心数的数量。这样可以充分利用 CPU 资源，避免线程过多导致频繁的上下文切换，提高任务执行效率。过多的线程数量可能会导致竞争，降低性能。
  2. I/O 密集型任务：对于 I/O 密集型任务，线程池的大小通常可以设置得更大一些，因为线程在等待 I/O 操作完成时并不会占用 CPU 资源，可以允许更多的线程同时执行。一般建议将线程池大小设置为 2 * CPU 核心数 或者更大，以确保能够充分利用 CPU 和 I/O 资源。

13.ForkJoin

ForkJoin 是 Java 中用于并行计算的框架，主要用于处理分治任务。它基于工作窃取（work-stealing）算法，可以将一个大任务拆分成多个小任务，并且在多个线程之间动态地调度这些任务，以实现并行计算的效果。

ForkJoin 框架的核心是 ForkJoinPool、ForkJoinTask 和 RecursiveTask（或 RecursiveAction）：

  1. ForkJoinPool：ForkJoinPool 是一个特殊的线程池，它使用工作窃取算法来提高并行任务的执行效率。ForkJoinPool 中的线程通过工作队列来存放任务，每个线程都有自己的工作队列，当自己的队列中没有任务时，会去其他线程的队列中窃取任务执行。

  2. ForkJoinTask：ForkJoinTask 是一个抽象类，用于表示可以并行执行的任务。它有两个重要的子类：RecursiveTask 和 RecursiveAction。RecursiveTask 用于有返回值的任务，而 RecursiveAction 用于没有返回值的任务。

  3. RecursiveTask：RecursiveTask 是一个抽象类，表示可以返回结果的任务。当一个任务需要拆分成多个子任务进行并行计算，并且最终需要合并子任务的结果时，可以使用 RecursiveTask。

  4. RecursiveAction：RecursiveAction 是一个抽象类，表示不需要返回结果的任务。当一个任务只需要执行某些操作，而不需要返回结果时，可以使用 RecursiveAction。

ForkJoin 框架适用于需要并行处理的分治任务，可以有效地利用多核处理器的优势，提高程序的性能。在使用 ForkJoin 框架时，需要注意合理设计任务的拆分和合并策略，以及避免出现死锁等并发问题。

14.异步调用

异步调用是一种编程模型，用于处理需要等待较长时间的操作，例如网络请求、文件读写、数据库查询等。在传统的同步调用中，调用者会等待操作完成后才能继续执行后续代码，而在异步调用中，调用者可以继续执行后续代码，而不必等待操作完成。

在异步调用中，通常会通过回调函数、Future/Promise 对象或者异步方法来处理操作完成后的结果。这样可以提高程序的响应性和并发性，避免阻塞主线程。异步调用常用于需要处理大量并发请求或者需要等待外部资源响应的场景，能够更有效地利用系统资源。

在 Java 中，可以使用 CompletableFuture、FutureTask、Callback 等机制来实现异步调用。在 JavaScript 中，可以使用 Promise、async/await 等机制来实现异步调用。异步调用在现代编程中非常常见，能够提高程序的性能和用户体验。

15.JMM

什么是JMM

JMM 是 Java 内存模型（Java Memory Model）的缩写，是 Java 虚拟机规范中定义的一种抽象的计算机内存模型，用于屏蔽不同硬件和操作系统的内存访问差异，确保在不同平台上对 Java 程序的并发访问都能得到一致的结果。

Java 内存模型规定了所有线程之间共享的主内存以及每个线程私有的工作内存之间的关系，主要包括以下几个方面的内容：

  1. 原子性（Atomicity）：对基本数据类型的读取和赋值操作具有原子性，即这些操作不会被中断。

  2. 可见性（Visibility）：一个线程修改的变量对其他线程是可见的，即一个线程修改的共享变量的值在其他线程中是可以看到的。

  3. 有序性（Ordering）：在单个线程内，程序的执行顺序按照代码的顺序执行。在多线程情况下，由于指令重排序的存在，可能会导致不同线程观察到的执行顺序不同。Java 内存模型通过 happens-before 原则来规定多线程之间操作的执行顺序。

关于JMM的一些同步约定

  1. 线程解锁前，必须将共享变量立刻刷新回主存。
  2. 进程加锁前，必须读取主存的最新值到工作内存中
  3. 加锁解锁都必须是同一把锁。

16. Volatile

  1. 可见性：被 volatile 修饰的变量在一个线程中被修改后，会立即更新到主内存中，其他线程读取该变量时会直接从主内存中获取最新值，保证了变量的可见性。
  2. 禁止重排序：volatile 修饰的变量禁止了指令重排序，保证了变量的修改顺序与代码中的顺序一致。
  3. 不保证原子性：volatile 修饰的变量对于单次读/写操作是原子的，但是对于多次操作（比如 i++）并不能保证原子性，如果需要保证原子性，可以使用 synchronized 关键字或者 java.util.concurrent.atomic 包中的原子类

如果不使用lock和synchronized，如何保证原子性？

使用原子类 Atomic，底层直接根操作系统挂钩，在内存中修改值。

指令重排

代处理器为了提高执行效率而进行的一种优化手段。处理器在执行指令时，并不一定会按照代码中的顺序依次执行，而是可能会重新排列指令的执行顺序，以尽量减少指令之间的相关性，提高并行度和性能。

volatile如何避免指令重排

volatile关键字实现禁止指令重排的机制是通过在编译器层面和运行时层面进行限制的。在编译器层面，编译器会禁止对volatile变量进行重排序优化，保证在执行顺序上不会将后面的操作排在前面。在运行时层面，JVM会在每次对volatile变量进行读写操作时，插入内存屏障指令（Memory Barrier），确保变量的写操作先于后续的读操作，从而保证了操作的顺序性。

18.CAS相关

什么是CAS

CAS（Compare and Swap，比较并交换）是一种用于实现多线程同步的原子操作。在CAS操作中，系统会先比较内存中某个位置的值与预期值是否相等，如果相等，则将该位置的值更新为新的值；如果不相等，则操作失败。整个操作是原子的，即在同一时刻只有一个线程能够成功执行该操作。

CAS操作通常用于实现非阻塞算法和并发数据结构，例如Java中的AtomicInteger、AtomicLong等类就是基于CAS操作实现的。CAS操作的优势在于它避免了传统的加锁操作所带来的性能损耗，因为它是基于硬件级别的原子操作，不需要像锁一样在用户态和内核态之间切换。

3、ABA问题

ABA问题（狸猫换太子）

一个线程读取了值之后，另外一个线程修改了这个值两次，最终改为了原来的值，第一个线程发现值没有变化，则不会发现其他线程修改过。

19.原子引用

解决ABA问题，引入原子引用，对应的是乐观锁思想

解决ABA问题的常见方法是使用带有版本号的原子引用，例如AtomicStampedReference类。这个类在比较时除了比较值外，还会比较一个版本号（stamp），只有当值和版本号都匹配时才进行修改。这样一来，即使值在过程中发生了变化，只要版本号不同，就能够避免误判。

2. 可重入锁

可重入锁（Reentrant Lock）是一种支持在同一个线程中多次获取同一把锁的锁机制。这意味着线程可以在持有锁的情况下再次获取该锁，而不会造成死锁。可重入锁是一种比较常见的锁实现方式，在Java中通过synchronized关键字和ReentrantLock类来实现。

在使用可重入锁时，同一个线程可以多次获取锁而不会被阻塞。线程每次成功获取锁后，需要在最后释放相同次数的锁。这样可以确保锁的所有权不会被其他线程所获取，从而避免了死锁的发生。

可重入锁的一个典型应用场景是递归函数。如果一个函数是递归的，并且这个函数需要获取锁来访问临界区资源，那么使用可重入锁可以确保同一线程在递归调用过程中可以多次获取锁而不会产生死锁。

总的来说，可重入锁是一种非常实用的锁机制，能够简化并发编程中的锁管理，并提高程序的可靠性和性能。

3.自旋锁（底层是CAS）

自旋锁是一种基于忙等待的锁机制，在尝试获取锁时，线程不会立即进入阻塞状态，而是循环检测锁的状态，直到获取到锁为止。自旋锁适用于锁被持有时间较短的情况，因为长时间的自旋会消耗CPU资源。

在Java中，AtomicInteger等原子类的compareAndSet()方法就是使用自旋锁实现的。当多个线程同时尝试修改同一个AtomicInteger的值时，如果没有竞争，compareAndSet()会立即成功；如果有竞争，线程会不断尝试，直到成功为止。

自旋锁的优点是避免了线程切换的开销，缺点是会消耗CPU资源，并且在高并发情况下可能会导致性能下降。因此，使用自旋锁时需要根据实际情况权衡。

4.死锁

死锁是指两个或多个线程（或进程）在执行过程中，因争夺资源而造成的一种互相等待的现象，导致它们都无法继续执行下去。简单来说，就是多个线程在互相等待对方释放资源，导致它们都无法继续执行下去。

死锁发生的必要条件包括：

  1. 互斥条件（资源只能被一个线程占用）、
  2. 请求与保持条件（一个线程持有了至少一个资源，并且正在请求另一个资源，但该资源被其他线程持有）、
  3. 不剥夺条件（线程不能强制占有其他线程的资源，只能自己释放）
  4. 循环等待条件（若干线程之间形成一种头尾相接的循环等待资源关系）。

死锁在多线程编程中是一个比较常见的问题，因为线程在获取资源时可能会发生竞争，如果不合理地使用锁或资源分配策略，就容易造成死锁。预防死锁的方法包括使用适当的锁策略、避免循环等待、按照固定的顺序获取锁等。
