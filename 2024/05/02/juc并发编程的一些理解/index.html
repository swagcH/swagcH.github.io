1.什么是JUC

JUC（Java Util Concurrency）是Java工具类库中的一部分，提供了在多线程环境下编写并发程序所需的工具和框架。JUC包含了一些并发编程的核心概念和工具，使得开发人员能够更轻松地编写高效、线程安全的并发程序。

2.进程和线程

  1. 进程：一个程序的集合。称为进程。

  2. 一个程序可以包含多个线程，至少包含一个线程！

  3. Java默认包含两个线程，main ，GC

  4. Java开启线程的方式，Thread，Runnable，Callable

  5. Java是无法自己开启线程的，本质上Java是通过操作c来开启线程

    ​ 并发，并行

    并发：多个线程操作同一个资源

    • cpu单核，模拟出多个线程，不停切换

    并行： 多个线程同时操作多个资源

    • ​ 多核cpu，多个线程同时执行

    并发编程的本质：充分利用cpu的资源

    线程状态

    ​ wait/sleep的区别

    1. 来自不同的类

      Wait =>object,sleep =>Thread

    2. 锁的释放

      wait会释放锁，sleep 不会释放锁！

    3. 使用的范围不同

      wait：必须在同步代码块中。sleep没有限制

    4. 唤醒方式

      wait：需要调用notify或者notifyAll唤醒，sleep自动唤醒，或者调用interrupt（）方法唤醒

Synchronized 和 Lock 区别

  1. synchronized关键字和Lock接口是Java中用于实现线程同步的两种不同方式，它们之间有以下几点区别：

    1. 使用方式：

      • synchronized是Java语言提供的关键字，可以直接用于方法或代码块中，不需要显式地创建锁对象。
      • Lock是一个接口，需要显式地创建锁对象，通常使用ReentrantLock类来实现。

    2. 灵活性：

      • synchronized是一种内置的锁机制，比较简单，但缺乏一些高级功能。
      • Lock接口提供了更多的功能，如可中断锁、超时获取锁、尝试非阻塞获取锁等。

    3. 等待锁的方式：

      • synchronized在获取不到锁时，线程会进入阻塞状态，直到获取到锁为止。
      • Lock在获取不到锁时，可以选择等待、放弃等待或者做其他处理。

    4. 释放锁的方式：

      • synchronized在方法执行完毕或代码块结束时会自动释放锁。
      • Lock需要手动调用unlock()方法来释放锁。

    5. 性能：

      • synchronized在JVM层面上对锁进行优化，因此在某些情况下性能可能比Lock更好。
      • Lock提供了更多的灵活性和控制，但由于是通过Java代码实现的，性能上可能会稍差。

    总的来说，synchronized适用于简单的线程同步需求，而Lock适用于对锁需要更精细控制和灵活性的场景。在选择使用时，需要根据具体情况来决定哪种方式更合适。

虚假唤醒问题

虚假唤醒（Spurious Wakeup）是指在多线程编程中，一个线程在没有被通知、没有收到信号的情况下，从等待状态（通常是wait()函数）中被唤醒，但此时并没有符合其等待条件的事件发生。这种情况可能会在某些多线程环境中发生，尽管并没有其他线程调用notify()或notifyAll()来通知等待线程，等待线程仍然被唤醒。

虚假唤醒可能会导致程序逻辑错误，因为等待线程被唤醒后可能会继续执行，但实际上并没有真正的事件发生。为了防止虚假唤醒，通常在使用wait()方法时，应该始终在循环中检查条件，并在条件不满足时继续等待，直到条件满足为止。例如：

Cow,读写分离

Copy-on-write（写时复制）是一种用于处理共享数据的技术，它可以解决写入时覆盖的问题。当多个进程或线程要访问同一个资源时，如果每个进程都复制一份资源的副本，会占用大量内存。相反，可以让所有进程共享同一个资源的副本，但只有在有进程要修改资源时才会复制资源，这就是写时复制的概念。

具体来说，当一个进程要修改资源时，系统会首先检查资源的引用计数。如果这个资源只被一个进程引用，那么该进程可以直接修改资源而无需复制。但如果资源被多个进程引用，系统就会将资源复制一份，然后让要修改资源的进程修改新复制的资源，而其他进程继续共享原始资源。这样一来，每个进程都可以独立地修改自己的资源副本，而不会影响其他进程。

通过写时复制，可以在多个进程之间共享资源的同时，避免了写入时的覆盖问题，提高了系统的性能和效率。

Map的解决方案

  1. Map map = new ConcurrentHashMap<>();
  2. Collections.synchronizedMap；

原理：

ConcurrentHashMap 是 Java 中线程安全的哈希表实现，它采用了一种不同于传统 HashMap 的机制来实现并发访问。主要原理如下：

  1. 分段锁（Segment）：ConcurrentHashMap 内部维护了一个 Segment 数组，每个 Segment 就是一个独立的哈希表，类似于将一个大的哈希表分成多个小的哈希表。每个 Segment 都相当于一个小的 HashMap，它们之间相互独立，可以独立进行并发操作。

  2. Hash 表的扩容：与 HashMap 类似，ConcurrentHashMap 也会在需要时对哈希表进行扩容。但不同的是，它只会对某个 Segment 进行扩容，而不是对整个哈希表进行扩容。这样可以减少扩容时的锁竞争，提高并发性能。

  3. put 操作的实现：当执行 put 操作时，ConcurrentHashMap 首先根据 key 的哈希值确定要放入哪个 Segment 中，然后在该 Segment 上加锁，进行插入操作。这样可以保证在放入元素时，其他线程可以继续访问其他 Segment 中的元素，不会受到阻塞。

  4. get 操作的实现：与 put 操作类似，get 操作也会根据 key 的哈希值确定要查找哪个 Segment，然后在该 Segment 上加锁，进行查找操作。这样可以保证在查找元素时，其他线程可以继续访问其他 Segment 中的元素，不会受到阻塞。

  5. 性能优化：ConcurrentHashMap 在实现上进行了多项性能优化，如使用了 volatile 修饰的数组元素来保证可见性，使用了 CAS（Compare and Swap）操作来保证原子性，以及使用了分段锁来减少锁竞争等。
